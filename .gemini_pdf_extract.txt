 
HACKATHON PROBLEM STATEMENT 
LLM Observability & AI Cost-Optimization 
Platform 
Build a web application that measures, visualizes, and optimizes how AI is used across 
an organization. 
Organized by: Aigenthix Technologies Private Limited 
Dates: 15/06/2026 - 16/06/2026 
1. Overview 
As organizations adopt large language models (LLMs) across products and internal tools, a 
recurring blind spot emerges: teams rarely know how much their AI actually costs, how fast it 
responds, or where the money is going. This hackathon asks participants to close that gap. 
The challenge is to build a web application for LLM observability — a platform that lets a user 
connect their own APIs, automatically surfaces the AI usage inside them, and renders the 
resulting KPIs and metrics on a live dashboard. On top of that observability layer, teams will 
build an AI optimization agent that reads the collected data and recommends concrete, 
actionable ways to cut LLM cost while keeping the system scalable. 
2. The Challenge 
Build a working web application that does three things: 
•​ Integrates any API and instruments it. Let a user register an API endpoint and route or 
wrap its LLM traffic so the platform can observe each call. 
•​ Visualizes AI KPIs in real time. Render charts and metrics — input/output tokens, 
response time, throughput, cost, error rates, and more — on an interactive dashboard. 
•​ Recommends optimizations through an agent. An AI agent analyzes the collected 
telemetry, identifies the biggest cost drivers, and proposes specific, scalable changes with 
projected savings. 
Important scoping note: “observe any API” only works when the AI usage is exposed 
somewhere. Section 4.3 defines the three capture patterns teams may use — read it before 
designing, as it is the single most common place teams get stuck. 
3. Objectives 
A strong submission should demonstrate the following outcomes: 
•​ A user can connect at least one LLM-backed API and see its activity reflected on the 
dashboard within minutes.

4. Functional Requirements 
4.1 The Observability Dashboard 
The dashboard is the heart of the product. A user should be able to add an API integration, then 
watch metrics populate as traffic flows through it. Charts should be filterable by time range, 
model, and endpoint, and should support drilling from an aggregate number down to individual 
requests (a request log with prompt, completion, tokens, latency, and cost per call).

4.3 Token & Cost Capture — the critical scoping decision 
You cannot point at an arbitrary REST endpoint and magically know its internal LLM token 
usage — that data has to be exposed somewhere. There are three capture patterns. Teams 
should state clearly which one(s) they support.

Pattern 
How it works 
Provider usage fields 
When the app calls an LLM provider directly, the response body 
already contains a usage object with input/output token counts. 
Trivial to log — works whenever the raw LLM response is visible. 
Proxy / gateway 
Route LLM traffic through a logging proxy that sits in the middle (the 
pattern behind tools like Helicone and LiteLLM). The integrated API 
points its base URL at the team’s proxy, which transparently logs 
tokens, latency, and cost. This is the real answer to “integrate 
anything” — and the most scalable architecture. 
Tokenizer estimation 
When only prompt and completion text are visible but no usage field 
is returned, estimate tokens with the model’s tokenizer (e.g. tiktoken 
for OpenAI-family models). Approximate, but the only option for 
opaque APIs. 
Recommendation: frame the challenge around the proxy/gateway or provider-usage approach. 
Teams that attempt to read tokens from a fully opaque third-party API without one of these 
mechanisms will not get accurate numbers. 
4.4 The AI Optimization Agent 
The agent should not give generic advice. It should read the team’s own collected telemetry, find 
the largest cost drivers, and propose specific changes with an estimated impact. A winning 
example of output: “Your summarization endpoint costs $X/day and 90% of calls are short — 
route it to a smaller model for an estimated 70% saving.” 
Cost-reduction levers the agent can reason about and recommend: 
•​ Model cascading / right-sizing — send easy queries to a cheaper/smaller model and 
escalate only the hard ones. 
•​ Prompt caching & semantic caching — reuse responses for identical or similar inputs. 
•​ Context reduction — trim prompts or use retrieval (RAG) instead of stuffing large context. 
•​ Batch processing — move non-realtime workloads to a batch API for a steep discount. 
•​ Output control — cap max output tokens and use structured outputs to avoid runaway 
generations. 
•​ Distillation / fine-tuning — replace a large model with a small fine-tuned one for 
repetitive, narrow tasks. 
5. Technical Guidance 
5.1 Scalable Architecture 
The “scalable approach” the brief asks for comes down to one principle: observability must 
never slow down the thing it observes. The recommended pattern is an asynchronous pipeline. 
•​ Non-blocking logging — capture telemetry on the call path without adding latency; 
fire-and-forget to a buffer.

•​ Buffer / queue — absorb spikes with a queue or stream (e.g. Redis Streams, Kafka). 
•​ Time-series / columnar store — persist events in a store built for analytics (ClickHouse is 
the de facto choice; Postgres or TimescaleDB is fine within a hackathon). 
•​ Aggregation + dashboard — query and roll up the data behind an API that feeds the 
charts. 
Optional bonus: adopting the OpenTelemetry GenAI semantic conventions makes the 
telemetry portable and is a strong signal of a production-minded design.
